# Personal Health Checker 项目总结

## 1. 项目定位

`Personal Health Checker` 是一个面向个人用户的 Web 健康追踪器 MVP。

当前版本的核心目标是：

- 让用户可以低成本记录每天的健康数据
- 用尽量清晰的页面结构呈现状态、趋势和目标
- 先跑通最核心的记录与回看主路径，再逐步打磨体验和商业化

当前 MVP 只聚焦 3 项指标：

- 睡眠
- 体重
- 饮水

当前明确不做：

- Apple Health / Health Connect 接入
- iOS / Android 原生端
- 社交功能
- 复杂 AI 建议
- 医疗诊断相关能力

## 2. 项目路径变化

这个项目并不是一开始就确定做 Web 的。

项目推进过程中，方向经历了几次关键收敛：

1. 最初讨论过 App Store 路线和穿戴设备接入。
2. 后来考虑过微信小程序和更低成本的启动方案。
3. 最终确定先做 `Web MVP`，目标是先验证产品主路径和用户体验，而不是一开始就做重平台、重硬件接入。

这个决策带来的好处是：

- 开发和迭代更快
- 成本更低
- 更容易先验证产品价值，而不是过早被平台和设备生态绑住

## 5. 当前技术实现

- `Next.js App Router`
- `React`
- `TypeScript`
- `Tailwind CSS`
- `Recharts`

### 5.2 后端

当前没有单独的 Express 服务，后端直接集成在 Next.js 内：

- `Route Handlers`
- `Prisma`
- `Zod`
- `bcryptjs`

### 5.3 数据库

- 数据库类型：`PostgreSQL`
- 当前接入：`Neon PostgreSQL`
- 通过 Vercel 集成资源连接

### 5.4 登录与会话

- 用户密码哈希后存库
- Session 存在数据库中
- 浏览器保存 `HttpOnly Cookie`

### 5.5 自动建表

为了避免空库导致线上注册/登录失败，项目里加了自动建表逻辑：

- 首次访问关键接口时，如果数据库里还没有表结构，会先自动补齐基础 schema

## 6. 当前数据模型

当前核心表包括：

- `User`
- `UserProfile`
- `Session`
- `DailyRecord`
- `Goal`
- `ProductEvent`

它们分别承担：

- `User`：登录主体
- `UserProfile`：昵称、时区、单位、提醒偏好
- `Session`：登录态
- `DailyRecord`：睡眠、体重、饮水每日记录
- `Goal`：三项健康目标
- `ProductEvent`：关键路径行为事件与页面访问观察

## 7. 当前注册逻辑

用户注册时，不只是“存一个邮箱”。

实际逻辑是：

1. 用户提交 `email + password + displayName`
2. 后端校验参数是否合法
3. 密码被哈希后写入数据库
4. 创建一条 `UserProfile`
5. 创建一条登录 Session
6. 给浏览器写入登录 Cookie

所以注册成功后，数据库里至少会新增：

- 一条 `User`
- 一条 `UserProfile`
- 一条 `Session`

## 9. 上线与环境演进

项目推进过程中，线上环境也经历了几次关键修复：

### 9.1 访客体验页上线

- 建立了无需登录的产品体验入口
- 用于快速查看界面结构和主路径

### 9.2 线上注册/登录修复

早期线上注册失败的原因不是前端，而是：

- Vercel 环境缺数据库变量
- 数据库为空库
- 预览环境受保护

后来完成了：

- 接入 Neon 数据库
- 切换到 Postgres adapter
- 自动建表
- 修复线上注册/登录流程

### 9.3 文案与产品感打磨

后续重点做了几轮产品表达收口：

- 去掉草稿感、导览感、解释感
- 强化可点击元素和静态信息块的视觉区分
- 修复页面切换时自动回到顶部的问题
- 将首页和 `/experience` 文案改得更短、更像成熟产品
